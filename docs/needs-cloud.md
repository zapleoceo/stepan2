# Облако потребностей лидов (needs cloud)

Виджет на странице отчётов (`/ui/reports/panel`, справа от воронки) — 3 колонки
**Боли · Цели · Выгоды**, ИИ-агрегация того, что Степан собрал в `lead.needs`
(`pains`/`jobs`/`gains`) по всем чатам. В каждой колонке — сущности по убыванию частоты
с визуальным баром веса.

## Архитектура — два слоя

**1. Классификация (ИИ, раз в сутки, инкрементально).**
[`aggregate_needs`](../app/worker/main.py) — крон в **17:00 UTC = 00:00 WIB (полночь
Джакарты)** по всем филиалам. [`classify_branch`](../app/modules/needs_cloud/service.py)
маппит фразы лида на **персистентную каноническую таксономию** (`need_entity`): модель
получает уже существующие категории и обязана переиспользовать их, новую заводит только
если фраза не подходит ни к одной. Обрабатываются только лиды, чей `needs` изменился
(`need_lead_state.needs_sha`) — отсюда дешевизна и **стабильность** (лейблы не прыгают
день-ко-дню). Результат — `lead_need_tag` (лид ↔ сущность).

**2. Виджет (без ИИ, мгновенно, любой диапазон).**
[`cloud_for`](../app/modules/needs_cloud/service.py) — `COUNT(DISTINCT lead)` по сущности с
фильтром `lead.created_at` в выбранном на странице диапазоне. Вес бара = доля от топ-сущности
колонки. Классификация дорогая, но делается раз на лида и переиспользуется всеми диапазонами.

## История

Каждую ночь [`write_snapshot`](../app/modules/needs_cloud/service.py) замораживает частоты
всех сущностей за день в `need_agg_snapshot` (уникально по `branch/kind/entity/дата`) — для
анализа эволюции облака во времени и сверки день-к-дню.

## Язык лейблов

Канонические лейблы — короткие **русские** строки (основной потребитель отчётов — владелец).
Модель маппит Bahasa/English-фразы лидов в русскую категорию.

## Таблицы

`need_entity` (таксономия) · `lead_need_tag` (лид↔сущность, счёт по диапазону) ·
`need_lead_state` (инкрементальность) · `need_agg_snapshot` (история). Миграция
`e7f8a9b0c1d2`.
