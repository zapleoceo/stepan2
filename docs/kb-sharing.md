# Общая база знаний между филиалами

Филиал может **не держать свою базу знаний**, а взять её из другого филиала. Два режима:

- **Связать (link)** — филиал читает персону / продукты / доки **живьём**
  из филиала-источника. Один источник правды: правишь источник — все связанные видят
  сразу. База связанного филиала read-only (правь источник).
- **Скопировать (copy)** — разовый клон базы другого филиала в этот. Дальше независимо
  (своя база, редактируется). Копирование снимает live-связь.

Чаты, лиды, воронка, сегменты, настройки — **всегда свои** у филиала. Общая только БЗ.

## Где в UI

Филиалы → выбрать филиал → форма:
- **«База знаний из филиала: [—своя— / другой]»** — селект в форме, сохраняется по «Save».
  Это режим **link**.
- **«Скопировать из: [филиал] [Скопировать]»** — режим **copy** (заменяет текущую базу).

## Как устроено (код)

- Колонка `branch.kb_source_branch_id` (nullable). Заполнена → филиал связан.
- `effective_kb_branch(session, branch_id)` (`app/modules/knowledge/source.py`) —
  возвращает источник, если связан, иначе сам филиал. Один хоп (источник не может быть
  сам связан — UI это запрещает).
- Все **чтения** KB резолвят эффективный филиал: reply (worker), followup, sim,
  call_failed, coach analyze, вкладка «База знаний». `KnowledgeService`
  строится с эффективным филиалом → персона/продукты/доки идут из источника.
- `copy_kb(session, dst, src)` — DELETE доков/продуктов dst + INSERT…SELECT из src.

## Пример: тестовый филиал на боевой базе

ClodeCouch (тест-филиал для sim) связан с Индонезией → тестирует **актуальную** боевую
базу без дубликата и без ручной синхронизации. Правки боевой базы Индонезии сразу видны
в симуляциях ClodeCouch.

## Ограничения

- Связь — один уровень (нельзя связать со связанным филиалом).
- Правка базы связанного филиала через его вкладку/коуч ложится на **источник** (это
  общая база). Для ясности правь источник напрямую.
