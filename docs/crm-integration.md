# Полная интеграция Stepan ↔ CRM (itstep MCP)

Источник CRM: `mcp.itstep.org/mcp/crm` (Streamable HTTP MCP, токен в URL), филиал
`jakarta` (id 435, IDR). Сохранено в супер-админ настройках: `crm_mcp_url`,
`crm_mcp_city_alias`. 34+ инструмента (динамический список).

## 1. Модель: единый лид

- **Ключ связки — телефон E.164.** `crm_client_search(phone)` → `id_uniq`;
  `crm_lead_add_event` принимает `phone` напрямую (id не нужен).
- **Разделение ответственности (важно):**
  - **Stepan = мессенджер** (IG/WA переписка, ведёт диалог).
  - **CRM = воронка/телефон/деньги** (звонки, расшифровки, договоры, оплаты, визиты,
    события, менеджеры).
  - Не дублируем — **джойним по телефону**. Один лид = чат в Степане + карточка в CRM.
- **Кэш связки:** `lead.crm_id_uniq` (+ время сверки), чтобы не искать каждый раз.

## 2. Хотелки → выполнимость на текущем MCP

| Задача | Сейчас | Что нужно |
|---|---|---|
| Читать/отвечать на сообщения лида **в CRM** | ❌ нет messaging-инструментов | **Виктор**: read/send message + «закреплён за Степаном»; ИЛИ признать, что мессенджер = Степан |
| События дальше по воронке после Степана | ⚠️ частично | calls/contracts/payments/leads_conversion есть; `client/history` для лидов даёт **500** — чинить Виктору |
| Связывать/анализировать чаты, объединять контекст Степан+CRM | ✅ чтение | наш агрегатор: `ai_summary` + `history` + `call_transcript` + `contracts` |
| «Когда заключится контракт» → в Мету | ⚠️ факт — да, прогноз даты — нет | факт `contract`/оплата → Meta CAPI; прогноз считает сам Степан |
| Писать исходы Степана в воронку CRM | ✅ уже можно | `crm_lead_add_event(phone, eventType)`, `crm_add_visit`, `crm_confirm_event` |

**Каталог событий `crm_lead_event_types` (jakarta):** `wait_call`, `thinking`,
`contract`, `reject`, `event`, `material`, `waiting_registration`, `next_enrollment`,
`interview`, `study_yes`. Это целевая карта для маппинга стадий Степана.

## 3. Архитектура (наша сторона)

1. **`CrmMcpClient`** — MCP-клиент к их серверу (Streamable HTTP, url из настроек,
   `cityAlias`, backoff, fail-open). Пакет `mcp` в проде уже есть.
2. **Identity-резолвер** — `lead.crm_id_uniq` по телефону (миграция + кэш).
3. **Агрегатор контекста CRM** — по phone/id тянет `ai_summary` + `history` +
   `call_transcript` + `contracts`/`payments`, склеивает с IG-диалогом → в
   `DecisionEngine` (Степан отвечает с учётом звонков менеджера) и в coach-analyze.
4. **Read-gate** — уже построен; подменить REST-`CrmReader` на `CrmMcpClient`
   (`crm_client_search` + `contracts`/`payments` → hold, если договор/оплата/актив).
5. **Write-back** — на переходах Степана: `ready`→`crm_lead_add_event(contract)`,
   отправка материалов→`material`, назначен фолоап→`wait_call`, запись на ивент→
   `crm_add_visit`/`event`. Идемпотентно.
6. **CRM→Meta петля** — факт `contract`/оплаты по нашему лиду → Meta CAPI
   `Purchase`/`Lead` (сумма договора, IDR; хэш PII). Reuse `meta_system_user_token`/
   pixel. Требует ad-attribution, снятую при создании лида.
7. **CRM-панель в окне чата** — рядом с IG-диалогом карточка CRM: найден?, события
   воронки, последняя расшифровка звонка, договоры/оплаты, AI-саммари.
8. **Каденс** — расширить крон `sync_crm` (поллинг активных/закреплённых лидов,
   rate-limit, fail-open).

## 4. Петля в Мету (замыкание рекламы)

`IG-реклама → лид (Степан снимает fbclid/источник) → Степан квалифицирует →
договор в CRM → CRM→Meta CAPI (сумма+валюта+хэш PII) → Meta оптимизирует на
покупателей`. Нужны: сумма/валюта договора из `contracts_list`, и click-id/лид-сорс,
сохранённый на лиде Степана при создании.

## 5. Что нужно доделать

### Сторона CRM (Виктор / itstep) — блокеры

1. **Мессенджер.** Держит ли CRM переписку лидов? Если да и нужен ответ Степана
   **в CRM** — нужны инструменты `messages_list`/`message_send` + понятие
   «закреплён за Степаном» (бот-агент). Если нет — фиксируем: мессенджер = Степан,
   а в CRM только исходы. **Это определяет весь пункт «читать/отвечать в CRM».**
2. **Починить `crm_client_history` (500)** для лид-записей (`id_client=null`).
3. **Список «закреплено за Степаном»** — фильтр/agent-id, чтобы Степан знал свой скоуп.
4. **Change-feed/webhook** «изменено с T» (иначе поллим всех); + лимиты запросов.
5. **Сигнал закрытия договора** — какое поле = «подписан»/«оплачен», сумма+валюта,
   и (если есть) прогноз даты закрытия по лиду.
6. **Идемпотентность/скоуп записи** — безопасно ли звать `crm_lead_add_event`
   повторно; токен per-branch или на все филиалы; лимиты записи.

### Наша сторона (Stepan) — строится без блокеров

1. `CrmMcpClient` + чтение настроек (url/alias уже сохранены).
2. `lead.crm_id_uniq` + резолвер (миграция).
3. Агрегатор контекста CRM → в reply-контекст и coach-analyze.
4. Read-gate на `CrmMcpClient` (гейт уже есть).
5. Write-back: маппинг стадий Степана → `crm_lead_add_event` + `crm_add_visit`.
6. CRM-панель в окне чата.
7. CRM→Meta мост (нужны атрибуция + сумма договора).
8. Каденс `sync_crm` + rate-limit/backoff + fail-open.

## 6. Фазы

- **Ф1 — read-gate: ✅ ПОСТРОЕН И ВКЛЮЧЁН (2026-07-17, branch 1).** `CrmMcpReader`
  (`app/adapters/crm_mcp.py`) читает карточку+историю по телефону; вердикт: договор →
  hold, успешный дозвон менеджера <72ч → hold, старый/недозвон — не держит. Точечная
  проверка перед отправкой + pull-синк с тайм-бюджетом. Fail-open.
- **Ф1b — спасение недозвонов: ✅ ПОСТРОЕНО И ВКЛЮЧЕНО (crm_rescue_enabled).**
  Ежечасно в рабочие часы: лог звонков CRM → телефоны, до которых не дозвонились →
  матч с чатами Степана → call_failed-реанимация («звонили, не дозвонились — давайте
  здесь»). ≤2 за тик, кулдаун 7 дней на лида, скипает занятых человеком/активные чаты.
  (`app/modules/crm/rescue.py`, крон `crm_rescue`).
- **Ф2 (write-back, обратимо):** `crm_lead_add_event` на исходах Степана + визиты.
  Частично: rescue журналирует; события contract/material/wait_call ещё не пишем.
- **Ф3 (Meta-петля):** договор/оплата → CAPI. Нужна атрибуция.
- **Ф4 (сообщения в CRM):** **блок на Викторе** (инструменты + закрепление). Только
  если мессенджер живёт в CRM.
- **Следующее (C):** напоминания о событиях — нужен от CRM список записей на события
  (пока есть только счётчик `CountRemindAboutEvent`).
