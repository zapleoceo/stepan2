# Атрибуция рекламы и авто-привязка продукта

Как лид, пришедший по рекламе, получает продукт в чате ещё до того, как Степан начнёт
квалификацию, и как этим управляет оператор.

## Что приходит на входе

Приватный IG-адаптер вытаскивает контекст рекламы из инбокса и кладёт его на тред
(`channel_thread`), см. `app/adapters/channels/transports.py`:

- `ad_id` — ID объявления в Meta Ads Manager (`ad_context_data.ad_id`);
- `ad_media_id` — IG media ID креатива (`professional_metadata.ad_ig_media_id`);
- `ad_preview_url` — CDN-превью креатива (часто пустой);
- `lead_source` — `story` | `ad_clicktomsg` | пусто.

Запись этих полей — `app/modules/leads/ingest.py:_store` (только на первом входящем).

## Мост в Marketing API: связка по media pk, НЕ по ad_id

Ключевой факт, на котором держится вся связка с реальными деньгами:

> **`channel_thread.ad_id` из приватного IG API непригоден для джойна с Marketing API.**
> Он лежит в другом id-пространстве: прямой `GET /{ad_id}` отдаёт `code 100` («объект не
> существует»), и ни один из наших ad_id не встречается среди объявлений аккаунта.
> Формат обманчиво похож (18 цифр, тот же префикс) — но это разные объекты.

Джойнить надо по **`ad_media_id`** (это IG media **pk**). Шорткод медиа — это тот же pk,
записанный в base64 алфавитом Instagram, а Marketing API отдаёт у креатива
`instagram_permalink_url` с этим шорткодом. Отсюда цепочка:

```
channel_thread.ad_media_id  (media pk)
  → pk_to_shortcode()                       app/modules/ads/bridge.py — чистая функция
  → adcreatives.instagram_permalink_url     → creative_id
  → ads.creative{id}                        → ad_id → adset → campaign
  → insights                                → расход, глубина переписки
```

Мост покрыт тестами на реальных парах pk↔шорткод (`tests/test_ads_bridge.py`) — они
офлайн и ловят молчаливую поломку алфавита.

### Покрытие: 38%, и упирается оно НЕ в мост

Измерено на проде (Джакарта, 1297 лидов с рекламы): **38%** сквозного покрытия
`лид → объявление`. Разбор по шагам цепочки:

| Шаг | Покрытие | Комментарий |
|---|---|---|
| media pk → креатив (`adcreatives`) | **96,8%** | мост работает почти идеально |
| креатив → объявление (`ads`) | **~38%** | **вот здесь теряется** |

То есть креатив в кабинете **есть**, а объявления, которое его использует, **нет**.
Проверено на медиа с 355 лидами (`DaSONsVsS3g` → креатив `1021918063871518`):

- ни одно объявление аккаунта не ссылается на этот `creative_id` — **0 совпадений**;
- по `effective_object_story_id` того же поста — тоже **0**;
- фильтр `ad.effective_status` со всеми статусами (`ARCHIVED`/`DELETED`/…) даёт
  1154 объявления против 1145 по умолчанию — **+9, погоды не делает**;
- в аккаунте 4499 креативов на 1154 объявления — большинство креативов осиротевшие.

> **Вывод: 62% лидов приходят с объявлений, которых в `act_1000480912055519` нет.**
> Владелец продукта утверждает, что реклама из кабинета не удаляется (только
> выключается) — значит эти объявления, скорее всего, крутятся из **другого рекламного
> аккаунта**. В бизнесе `949920286532207` их 34 (вся сеть IT STEP) + 1 собственный, а наш
> System User назначен только на Джакарту (`me/adaccounts` → ровно один аккаунт).
> **Открытый вопрос к агентству: из какого кабинета реально крутится Джакарта.**

Историческая заметка, чтобы не наступить снова: цифра **96,8% какое-то время
фигурировала как сквозное покрытие — это была ошибка измерения** (мерили
media→креатив, а не media→объявление). Сквозное — 38%. Процент всегда печатается на
самой панели, поэтому расхождение видно оператору, а не прячется в доке.

Через `adcreatives` покрытие заметно выше, чем через вложенный
`ads?fields=creative{instagram_permalink_url}` (измерено: 3414 permalink'ов из 4499
креативов против 946 из 1145 ads) — поэтому рёбра обходятся **раздельно** и джойнятся
локально по `creative_id`.

## Маппинг реклама → продукт

Имя/кампания объявления теперь известны программно (см. мост выше), но **какой продукт
рекламирует объявление** — по-прежнему решает оператор. Источник истины —
**оператор-заданная таблица** `ad_product_map` (`ad_id → product_slug`, per-branch), сервис
`app/modules/ads/mapping.py`.

- `product_for_ad(ad_id)` — какой продукт рекламирует это объявление;
- `upsert / clear` — правки оператора;
- `suggest_from_history()` — **подсказка** для UI: самый частый `product_slug` среди
  прошлых тредов этой рекламы. Никогда не пишется в таблицу автоматически — это
  самоусиливающийся сигнал (отражает прошлые догадки Степана), только префилл.

## Провенанс продукта (`channel_thread.product_source`)

Продукт на треде может прийти из трёх источников; колонка `product_source` хранит,
откуда именно, и это гейтит переопределение:

| Источник | Значение | Кто ставит |
|---|---|---|
| Реклама | `ad` | ingest: `product_hint` из пейлоада или `ad_product_map` |
| Модель | `model` | `ReplyService._apply_decision` при уверенной квалификации |
| Менеджер | `manager` | ручная смена в UI чата (`_routes_chat.py`) |

Правило переопределения (`app/modules/conversation/reply.py:_apply_decision`): модель
может переквалифицировать продукт, унаследованный от рекламы (`ad`) или от своего же
прошлого хода (`model`), но **никогда не перебивает ручной выбор менеджера** (`manager`).
Так реклама даёт стартовый дефолт, Степан уточняет по диалогу, а решение человека —
финально.

## Редактор в отчётах

Таблица «Воронка по рекламе» (`/ui/reports`) при выбранном **одном** филиале рисует на
каждую рекламу:

- **Ячейка рекламы** — `<details>`-меню (чистый HTML, без JS): «Открыть чаты этой рекламы»
  (→ `/ui/inbox?ad_id=…`, список чатов фильтруется по `channel_thread.ad_id`) и «Открыть
  объявление в FB». FB-ссылка (`_fb_ad_url`) ведёт в **публичную Библиотеку рекламы**
  (`facebook.com/ads/library/?id=<ad_id>`), а не в Ads Manager. Причина (проверено живьём):
  `ad_context_data.ad_id`, который отдаёт Instagram, почти никогда не лежит в настроенном
  рекламном аккаунте (`fb_account_id`) — эти лид-объявления крутятся под другим (агентским)
  аккаунтом, но публикуются с той же FB-страницы. Поэтому Ads Manager (фильтр по аккаунту)
  показывал «не найдено», хотя объявление живо. Библиотека резолвит объявление по одному ID,
  независимо от аккаунта, и всегда открывает реальное живое объявление (креатив, текст,
  статус, рекламодатель) — только просмотр, без расходов/результатов. Рядом иконка 📷 —
  ссылка на IG-пост креатива (`ig_post_url(ad_media_id)`); при наведении — превью креатива.
- **Ховер-превью креатива** — Marketing API мы не читаем, поэтому превью тянется из
  публичного IG-пермалинка: `_ig_preview.og_image_for_media` достаёт `og:image` со страницы
  `/p/<code>/` (кэш в памяти, креативы неизменяемы), а `GET /ui/ig-preview/{media_id}`
  проксирует байты картинки **со своего домена** — так обходятся hotlink/referer/CORS
  ограничения cdninstagram. Тёмные/непубличные посты просто не дают превью (404 → ховер
  прячется).
- **Колонка «Продукт»** — `<select>`, который на `change` шлёт `POST /ui/ads/{ad_id}/product`
  и апсертит `ad_product_map`. Если реклама ещё не размечена, показывается чип ⚡ с
  автоподсказкой из истории (`suggest_from_history`) — клик применяет её.

Рендер ячейки (`admap_cell_inner`) переиспользуется и в таблице, и в ответе POST-роута,
чтобы htmx-своп подставлял идентичный HTML. Колонка и подсказки показываются только для
одного филиала (`_ad_editor_data` возвращает пусто при мультифилиальном отчёте).

**Сортировка и фильтры** — клиентские (таблица маленькая, без серверных round-trip'ов),
JS едет инлайном во фрагменте (`_AD_FUNNEL_JS`), обработчики через `on*`-атрибуты, так что
повторный htmx-своп их просто переопределяет. Клик по заголовку сортирует (числовые колонки
— по числу, текстовые — по строке, повторный клик меняет направление). Под заголовками —
строка фильтров по колонкам: подстрока по `ad_id`, `≥`-порог по числовым колонкам, дропдаун
по продукту. Значение ячейки для сортировки/фильтра берётся из `<select>` продукта →
`<summary>` с ad_id → текста ячейки, поэтому интерактивные колонки тоже сортируются.

- **Числа-колонки как ссылки** — «Всего / В работе / Закрытые / Спящие» кликабельны и ведут
  в инбокс, отфильтрованный по этой рекламе И по той же группе стадий, что стоит за счётчиком
  (`_count_cell` → `/ui/inbox?ad_id=…&grp=pipeline|won|dormant`; «Всего» — без `grp`). Наборы
  стадий берутся из `AD_FUNNEL_GROUPS` (`_query.py`), общих со счётчиками `fetch_ad_funnel`,
  чтобы число в отчёте и длина списка чатов совпадали. Ссылки не мешают сортировке/фильтру
  колонок — текст ячейки остаётся числом.

Фильтр чатов по рекламе: `threads_partial(ad_id=…, grp=…)` добавляет `ct.ad_id = :ad_id` и, при
наличии группы, `l.stage IN (…)`; шелл инбокса рисует дисмиссабл-чип (реклама + метка группы)
со ссылкой назад на нефильтрованный список.

## Визуализация воронки (страница «Отчёты»)

Три блока рисуются server-side как inline-SVG (без CDN/JS, чтобы переживать htmx-своп
фрагмента) в `app/api/_ui_panels.py`; все скоупятся тем же окном, что и KPI — по
`lead.created_at` (старт диалога) + фильтру филиалов:

- **Сегменты лидов** — `_segment_tree_html`: **две оси**. Классификация лида разложена на
  `lead_type` (намерение/температура: `hot|warm|cold|no_budget|non_target|unclear`) и
  `audience` (кто это: `adult|student`) — ортогонально, поэтому школьник может быть ещё и
  «горячим» (раньше `student` занимал единственный слот `lead_type` и стирал температуру →
  won% школьников всегда 0). Рендер — **под-дерево на каждую аудиторию** в порядке «Взрослые» →
  «Не определён» → «Школьники» (`_AUD_ORDER`), у каждого свой корень с total, ветвящийся в
  сегменты, **отсортированные по вероятности успеха** (won/count по убыванию, добор по объёму).
  Блок «Не определён» — лиды, у кого `audience` ещё NULL: их НЕ приписываем к взрослым, они
  наполняются во «Взрослые»/«Школьники» по мере классификации вживую. Толщина связи ∝ объёму,
  на листе count · доля% · won%. Данные — `fetch_segment_dist`, строки
  `(audience, lead_type, total, won)` c `coalesce(audience,'unknown')`,
  `coalesce(lead_type,'unclear')`, won = стадии `ready|handed_off`; **потребители индексируют
  total как `s[2]`**. У каждого листа hover-`<title>` (`seg.tip` + `segdesc.{type}`) с точным
  критерием — дословно из блока LEAD TYPE в `prompt.py`. Лист — SVG-ссылка на
  `/ui/inbox?lead_type=<type>&audience=<aud>` — **обе оси**, поэтому «тёплые школьники»
  открывают ровно тех, а не всех тёплых (для `unclear`/`unknown` фильтр ловит и `NULL`); узел
  воронки — ссылка на `/ui/inbox?stage=<stage>`. `threads_partial`/`inbox` принимают `lead_type`
  и `audience`; шелл рисует дисмиссабл-чип (`inbox.seg_filter`, «Школьники · тёплые»). Фильтр
  переживает открытие чата и фоновое обновление (несётся в URL чата и `#tl`-поллинг). Если
  аудитория одна — корень остаётся
  «Всего лидов N» без разбивки.
- **Воронка (путь лидов)** — `_funnel_flow_html`: Sankey-поток реального пути
  `вход → … → выход`, восстановленный из аудита переходов `stage_event` (`fetch_stage_flow`,
  строки `from_stage,to_stage,count`, self-переходы отфильтрованы). Счёт —
  `count(distinct lead_id)` на ребро (не сырые события): лид логирует несколько переходов, а
  миграция истории с S1 часть продублировала, поэтому сырые события завышают поток; уникальные
  лиды делают толщину = «сколько людей прошло», сопоставимо с деревом. Число/высота узла —
  `fetch_stage_reach` (distinct лидов, коснувшихся стадии как `from` или `to`) = реальный
  headcount ≤ всего лидов; НЕ сумма входящих рёбер (она мульти-считает лида на нескольких
  рёбрах, из-за чего бар «вход» мог оказаться выше всей базы). Стартовый узел `new` подписан
  «вход» (`flow.entry`) — точка входа (первое сообщение), а не текущий остаток в стадии `new`.
  Спайн-стадии — верхняя
  дорожка, терминалы `dormant`/`manager` — нижняя; толщина связи ∝ distinct-лидам перехода,
  цвет — по целевой стадии, back-edges
  (напр. `presenting→qualifying`) гнутся — видно откаты и отвалы, не только happy path. У
  каждого узла hover-`<title>` с определением стадии (`sdesc.{stage}`, для входа
  `flow.entry_desc`). Отдельный пунктирный узел «без движения» (`flow.stuck`) =
  `total_leads − reach["*"]` (все лиды окна минус те, у кого есть хоть один переход): лиды,
  что написали, но ещё не сдвинулись — так «вход + без движения = всего лидов». `total_leads`
  = сумма сегментов; `reach["*"]` — distinct-лиды с любым переходом (спец-ключ из
  `fetch_stage_reach`). Если за окно истории переходов нет, блок пуст и `reports_panel_html`
  откатывается на старую линейную `_funnel_line_html`.
- **Пресеты периода** — `_QUICK_RANGES` (в `_ui_panels.py` для кнопок и в `_routes_admin.py`
  для `timedelta`): 1h/2h/4h/8h/12h/24h/7d/30d/60d/90d/весь. Держать оба списка в синхроне.

## Переклассификация лидов (`lead_type` + `audience`)

`lead_type` и `audience` живьём проставляются только как побочный эффект решения при ответе
(`ReplyService._apply_decision`), поэтому у исторических лидов они NULL (→ «не ясно» / «Взрослые»
в дереве). Миграция (ныне alembic-ревизия `e3f4a5b6c8d9`, поглотившая ручной
`031_lead_audience.sql`) перенесла существующих `lead_type='student'` на
`audience='student'` и сбросила их `lead_type` в `unclear` — температура восстановится на
следующем ответе в диалоге.

Бэкофилл — `scripts/reclassify_leads.py`: за ОДИН вызов брокера на лида переопределяет
стадию воронки + `lead_type` + `audience` уже по двух-осевой модели (relabel-only: пишет
`lead.stage` + аудит `stage_event`, `lead.lead_type`, `lead.audience` и ничего больше; не шлёт
ответов и не делает hand-off; ready/handed_off/manager никогда не назначает). Идемпотентно и
резюмируемо (каждый лид — своя транзакция), флаги `--branch/--limit/--dry`. Запуск в контейнере:
`docker exec -e PYTHONPATH=/app stepan2-worker python -m scripts.reclassify_leads`.
(Старый одно-осевой `reclassify_lead_types.py` удалён 2026-07-14 — он возвращал `student`
в `lead_type` в обход оси `audience`; преемник выше делает то же и больше, корректно.)

## Синхронизация с Meta: две таблицы с разной семантикой

Данные Marketing API делятся на два типа с **противоположными** требованиями к обновлению,
и именно поэтому «фоновая задача» здесь не означает «устаревшие цифры».

### `ad_creative_map` — неизменяемая карта

Permalink креатива не меняется, значит связь `media_pk → ad` верна навсегда. Ре-синка нет:
карта только **дополняется**. Обход Graph запускается, только если есть лид с неизвестным
`media_pk` (`repository.unmapped_media_pks`), и **обрывается сразу**, как только все искомые
шорткоды найдены. В установившемся режиме — **ноль запросов к Graph**.

### `ad_insight_daily` — скользящее окно

Одна строка на (объявление, день) → любой диапазон дат в отчёте это локальный `SUM`, без
похода в Graph. Тянем только объявления, **откуда реально есть наши лиды** (~50, не 1145).
Окно — `INSIGHT_WINDOW_DAYS = 14`: Meta правит атрибуцию ~7 дней, старше — заморожено.

Обновление — **delete+insert**, не upsert: Meta может вовсе убрать день при пересчёте
атрибуции, и upsert оставил бы протухшую строку навсегда.

### Rate limit — это норма, а не исключение

Рекламный аккаунт отдаёт `code 80004` («too many calls to this ad-account») после серии
страниц, и throttle действует **на весь аккаунт**. Поймали живьём при разработке. Поэтому:

- `MetaAdsRateLimited` несёт курсор (`next_url`) для возобновления;
- троттлинг **не коммитит частичные инсайты** — иначе расход молча занизится;
- крон `sync_ads` намеренно редкий (`minute={3,23,43}`) — чаще незачем, Meta сама лагает 7 дней;
- карта и инсайты — в **разных транзакциях**: троттл инсайтов не должен откатывать уже
  найденную (дорогую) строку карты.

Задачи: `sync_ads` → `sync_ads_branch` в `app/worker/main.py` (per-branch fan-out —
филиалы независимы). Не гейтится kill-switch'ем: это read-only отчётность, лиду ничего
не отправляется, поэтому оператор, остановивший бота, всё равно видит свои цифры.

### Свежесть на виду

`ad_insight_daily.synced_at` показывается в шапке блока («данные на 14:35»), а процент
связанных с рекламой лидов — рядом с таблицей. Таблица расхода, молча теряющая треть лидов,
читается как полная и получает незаслуженное доверие — поэтому покрытие всегда на лице.

## Что НЕ делается

- **Пиксель/CAPI по стадиям.** Событие в пиксель Meta должна с кем-то сопоставить.
  `build_event` умеет только хэш телефона, а у чат-лидов его нет (на `presenting` — 9 из 373).
  Для чат-лида нужен `ctwa_clid` из реферала рекламы — он приходит только в официальном
  Meta-канале (App Review), а через instagrapi его нет в принципе.
- **Per-lead атрибуция для не-IG плейсментов.** Креатив без `instagram_permalink_url`
  (сторис, не-IG размещения) в карту не попадает — такое медиа остаётся несвязанным и
  честно вычитается из процента покрытия.
