# Атрибуция рекламы и авто-привязка продукта

Как лид, пришедший по рекламе, получает продукт в чате ещё до того, как Степан начнёт
квалификацию, и как этим управляет оператор.

## Что приходит на входе

Приватный IG-адаптер вытаскивает контекст рекламы из инбокса и кладёт его на тред
(`channel_thread`), см. `app/adapters/channels/transports.py`:

- `ad_id` — ID объявления в Meta Ads Manager (`ad_context_data.ad_id`);
- `ad_media_id` — IG media ID креатива (`professional_metadata.ad_ig_media_id`);
- `ad_preview_url` — CDN-превью креатива (часто пустой);
- `lead_source` — `story` | `ad_clicktomsg` | пусто.

Запись этих полей — `app/modules/leads/ingest.py:_store` (только на первом входящем).

## Маппинг реклама → продукт

Meta Marketing API мы не читаем, поэтому имя/кампанию объявления программно не знаем.
Источник истины — **оператор-заданная таблица** `ad_product_map` (`ad_id → product_slug`,
per-branch), сервис `app/modules/ads/mapping.py`.

- `product_for_ad(ad_id)` — какой продукт рекламирует это объявление;
- `upsert / clear` — правки оператора;
- `suggest_from_history()` — **подсказка** для UI: самый частый `product_slug` среди
  прошлых тредов этой рекламы. Никогда не пишется в таблицу автоматически — это
  самоусиливающийся сигнал (отражает прошлые догадки Степана), только префилл.

## Провенанс продукта (`channel_thread.product_source`)

Продукт на треде может прийти из трёх источников; колонка `product_source` хранит,
откуда именно, и это гейтит переопределение:

| Источник | Значение | Кто ставит |
|---|---|---|
| Реклама | `ad` | ingest: `product_hint` из пейлоада или `ad_product_map` |
| Модель | `model` | `ReplyService._apply_decision` при уверенной квалификации |
| Менеджер | `manager` | ручная смена в UI чата (`_routes_chat.py`) |

Правило переопределения (`app/modules/conversation/reply.py:_apply_decision`): модель
может переквалифицировать продукт, унаследованный от рекламы (`ad`) или от своего же
прошлого хода (`model`), но **никогда не перебивает ручной выбор менеджера** (`manager`).
Так реклама даёт стартовый дефолт, Степан уточняет по диалогу, а решение человека —
финально.

## Редактор в отчётах

Таблица «Воронка по рекламе» (`/ui/reports`) при выбранном **одном** филиале рисует на
каждую рекламу:

- **Ячейка рекламы** — `<details>`-меню (чистый HTML, без JS): «Открыть чаты этой рекламы»
  (→ `/ui/inbox?ad_id=…`, список чатов фильтруется по `channel_thread.ad_id`) и «Открыть
  объявление в FB». FB-ссылка (`_fb_ad_url`) ведёт в **публичную Библиотеку рекламы**
  (`facebook.com/ads/library/?id=<ad_id>`), а не в Ads Manager. Причина (проверено живьём):
  `ad_context_data.ad_id`, который отдаёт Instagram, почти никогда не лежит в настроенном
  рекламном аккаунте (`fb_account_id`) — эти лид-объявления крутятся под другим (агентским)
  аккаунтом, но публикуются с той же FB-страницы. Поэтому Ads Manager (фильтр по аккаунту)
  показывал «не найдено», хотя объявление живо. Библиотека резолвит объявление по одному ID,
  независимо от аккаунта, и всегда открывает реальное живое объявление (креатив, текст,
  статус, рекламодатель) — только просмотр, без расходов/результатов. Рядом иконка 📷 —
  ссылка на IG-пост креатива (`ig_post_url(ad_media_id)`); при наведении — превью креатива.
- **Ховер-превью креатива** — Marketing API мы не читаем, поэтому превью тянется из
  публичного IG-пермалинка: `_ig_preview.og_image_for_media` достаёт `og:image` со страницы
  `/p/<code>/` (кэш в памяти, креативы неизменяемы), а `GET /ui/ig-preview/{media_id}`
  проксирует байты картинки **со своего домена** — так обходятся hotlink/referer/CORS
  ограничения cdninstagram. Тёмные/непубличные посты просто не дают превью (404 → ховер
  прячется).
- **Колонка «Продукт»** — `<select>`, который на `change` шлёт `POST /ui/ads/{ad_id}/product`
  и апсертит `ad_product_map`. Если реклама ещё не размечена, показывается чип ⚡ с
  автоподсказкой из истории (`suggest_from_history`) — клик применяет её.

Рендер ячейки (`admap_cell_inner`) переиспользуется и в таблице, и в ответе POST-роута,
чтобы htmx-своп подставлял идентичный HTML. Колонка и подсказки показываются только для
одного филиала (`_ad_editor_data` возвращает пусто при мультифилиальном отчёте).

**Сортировка и фильтры** — клиентские (таблица маленькая, без серверных round-trip'ов),
JS едет инлайном во фрагменте (`_AD_FUNNEL_JS`), обработчики через `on*`-атрибуты, так что
повторный htmx-своп их просто переопределяет. Клик по заголовку сортирует (числовые колонки
— по числу, текстовые — по строке, повторный клик меняет направление). Под заголовками —
строка фильтров по колонкам: подстрока по `ad_id`, `≥`-порог по числовым колонкам, дропдаун
по продукту. Значение ячейки для сортировки/фильтра берётся из `<select>` продукта →
`<summary>` с ad_id → текста ячейки, поэтому интерактивные колонки тоже сортируются.

- **Числа-колонки как ссылки** — «Всего / В работе / Закрытые / Спящие» кликабельны и ведут
  в инбокс, отфильтрованный по этой рекламе И по той же группе стадий, что стоит за счётчиком
  (`_count_cell` → `/ui/inbox?ad_id=…&grp=pipeline|won|dormant`; «Всего» — без `grp`). Наборы
  стадий берутся из `AD_FUNNEL_GROUPS` (`_query.py`), общих со счётчиками `fetch_ad_funnel`,
  чтобы число в отчёте и длина списка чатов совпадали. Ссылки не мешают сортировке/фильтру
  колонок — текст ячейки остаётся числом.

Фильтр чатов по рекламе: `threads_partial(ad_id=…, grp=…)` добавляет `ct.ad_id = :ad_id` и, при
наличии группы, `l.stage IN (…)`; шелл инбокса рисует дисмиссабл-чип (реклама + метка группы)
со ссылкой назад на нефильтрованный список.

## Визуализация воронки (страница «Отчёты»)

Три блока рисуются server-side как inline-SVG (без CDN/JS, чтобы переживать htmx-своп
фрагмента) в `app/api/_ui_panels.py`; все скоупятся тем же окном, что и KPI — по
`lead.created_at` (старт диалога) + фильтру филиалов:

- **Сегменты лидов** — `_segment_tree_html`: дерево «Всего лидов N» → лист на каждый
  `lead_type`, толщина связи ∝ объёму, на листе count · доля% · won%. Данные —
  `fetch_segment_dist` (`coalesce(lead_type,'unclear')`, won = стадии `ready|handed_off`).
- **Воронка (путь лидов)** — `_funnel_flow_html`: Sankey-поток реального пути
  `вход → … → выход`, восстановленный из аудита переходов `stage_event` (`fetch_stage_flow`,
  строки `from_stage,to_stage,count`, self-переходы отфильтрованы). Счёт —
  `count(distinct lead_id)` на ребро (не сырые события): лид логирует несколько переходов, а
  миграция истории с S1 часть продублировала, поэтому сырые события завышают поток; уникальные
  лиды делают толщину = «сколько людей прошло», сопоставимо с деревом. Число/высота узла —
  `fetch_stage_reach` (distinct лидов, коснувшихся стадии как `from` или `to`) = реальный
  headcount ≤ всего лидов; НЕ сумма входящих рёбер (она мульти-считает лида на нескольких
  рёбрах, из-за чего бар «вход» мог оказаться выше всей базы). Стартовый узел `new` подписан
  «вход» (`flow.entry`) — точка входа (первое сообщение), а не текущий остаток в стадии `new`.
  Спайн-стадии — верхняя
  дорожка, терминалы `dormant`/`manager` — нижняя; толщина связи ∝ distinct-лидам перехода,
  цвет — по целевой стадии, back-edges
  (напр. `presenting→qualifying`) гнутся — видно откаты и отвалы, не только happy path. Если
  за окно истории переходов нет, блок пуст и `reports_panel_html` откатывается на старую
  линейную `_funnel_line_html`.
- **Пресеты периода** — `_QUICK_RANGES` (в `_ui_panels.py` для кнопок и в `_routes_admin.py`
  для `timedelta`): 1h/2h/4h/8h/12h/24h/7d/30d/60d/90d/весь. Держать оба списка в синхроне.

## Переклассификация лидов (`lead_type`)

`lead_type` живьём проставляется только как побочный эффект решения при ответе
(`ReplyService._apply_decision`), поэтому у исторических лидов он NULL (→ «не ясно» в дереве).
`scripts/reclassify_lead_types.py` — one-off бэкофилл по образцу `rederive_needs.py`: берёт
ТОЛЬКО сообщения лида (`direction='in'`), гонит через брокер (`chat:smart`) с теми же 7
определениями типов, что и живой промпт (скопированы в скрипт дословно), пишет `lead.lead_type`.
Идемпотентно и резюмируемо (каждый лид — своя транзакция), флаги `--branch/--limit/--dry`.
Запуск в контейнере: `docker exec -e PYTHONPATH=/app stepan2-worker python -m
scripts.reclassify_lead_types`.
