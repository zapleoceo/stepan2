# Reply-пайплайн: как устроено ведение лида

Один ход бота = один проход `ReplyService.decide()` ([reply.py](../app/modules/conversation/reply.py)).
Слои по порядку; каждый живёт в своём модуле (SRP):

| Слой | Модуль | Ответственность |
|---|---|---|
| 1. Контекст | `engine.py` | Собрать досье: персона, KB (RAG + always-доки), карточка продукта, история, needs-профиль |
| 2. Ситуация | `situations.py` | **Определить особый момент хода** и вколоть ОДИН короткий нудж (см. ниже) |
| 3. Решение | `prompt.py` (+ брокер) | LLM возвращает строгий JSON: текст, стадия, needs, флаги |
| 4. Валидация needs | `needs.py` | Дедуп near-фраз; только слова лида (`lead_grounded`); боль ≠ вопрос |
| 5. Guard | `guard.py` | Анти-выдумки: URL/цены/запреты KB (`NEVER/jangan`), verify → regen → safe-fallback |
| 6. Гейты стадий | `reply._stage_for` | Код, не модель, решает стадию: презентация только при боли+ценности (`captured()`), ready только с телефоном, human-led стадии неприкосновенны |
| 7. Отправка | `outbox.py` | Капы/тихие часы/soft-block-backoff; фолоапы — `followup.py` |

## Ситуативный слой (`situations.py`) — почему он существует

Статический промпт (~26k) несёт все правила продаж, но живые аудиты (100 + 169 диалогов,
07-2026) показали: модель следует им **ненадёжно** на таком масштабе. Надёжный паттерн —
детерминированный детектор + короткая инструкция в конкретный ход. Все детекторы, нуджи и
их приоритеты живут в одном модуле, выбор — одна функция `pick_nudge()` (раньше цепочка
была размазана по reply.py — две параллельные правки однажды определили один нудж дважды).

Приоритет (первое совпадение, максимум один нудж на ход):

```
non-target        модель уже классифицировала лида ранее → вежливо закруглиться
ad-opener         лид ещё НЕ сказал ни слова своего (только кнопка рекламы) →
                  тёплое первое сообщение: кто мы + крючок + вопрос с цифрами-вариантами;
                  НИКОГДА не цена (текст кнопки «…dan biaya» — слова рекламы, не лида)
unseen-media      прислал нечитаемое (шаринг/войс/фото без распознавания) → честно
                  «не открывается», не выдумывать тему
minor             школьник ИЛИ родитель о ребёнке → платит родитель; школьнику не
                  давить ценой/DP, родителю продавать напрямую
soft-no           вежливое «не сейчас» (nanti/pikir dulu/insyaallah/tanya keluarga) →
                  признать, не давить, максимум один лёгкий шаг
answer-first      прямой вопрос своими словами → ответить фактом В ЭТОМ ходе;
                  нет в KB → честно + needs_manager
low-budget        нет денег/безработный/«mahal» → дешёвый вход первым, не полный прайс
need-payoff       боль есть, ценность нет → спросить желаемый результат до питча
                  (действует ТОЛЬКО до discovery-cap — согласован с гейтом стадий)
discovery-cap     >4 ходов без needs → перестать допрашивать, презентовать что есть
```

Комбо (конфликт правил решается «и-и», не «или»):
- **soft-no + вопрос** («nanti dulu… tapi berapa harganya?») → ответить кратко, затем
  отпустить без питча. Ни неотвеченного вопроса (самая дорогая утечка), ни дожима.
- **вопрос + нет денег** («ga ada modal, berapa biayanya?») → честная цена + дешёвая
  ступень рядом, без DP.

Плюс одноразовый **need-payoff-редрафт** в reply.py: боль всплыла в ЭТОМ ходе (нудж по
сохранённому профилю сработать не мог) → черновик пересобирается один раз, чтобы первая
боль не получила в ответ прайс.

## Фолоапы (`followup.py`)

Таймер от последнего сообщения бота; смена угла обязательна; near-дубли режутся дважды
(до guard и после), сухой повтор сжигает шаг расписания; исчерпание → dormant.
Лиду, который **ни разу не написал своими словами** (только клик по рекламе), фолоап
не имеет права называть цену/расписание — те же правила, что у первого сообщения
(`FOLLOWUP_SILENT_CLICKER_EXTRA`; инцидент 3926: первый же фолоап вывалил прайс молчуну).
`needs_manager` из фолоапа алертит **только** если лид говорил своими словами, его
последнее сообщение содержит вопрос и это не текст рекламной кнопки (в ней есть слово
«biaya» — тред 3926 поднял фантомный «Berapa biaya?»), и один раз за молчание
(инциденты 3072/2532).

## Регрессия

Любая правка промпта/KB/пайплайна → sim на ClodeCouch (branch 8) по
[dialogue-qa-checklist.md](dialogue-qa-checklist.md); детекторы/приоритеты покрыты
юнитами (`tests/test_situational_nudges.py` — включая комбо).

## Известные незакрытые хвосты

- Soft-no лид всё равно получает фолоапы по расписанию (нет снуза на названный срок).
- Текст-хелперы (`_split_bubbles`, `_most_similar_prior`) ещё в reply.py; followup
  импортирует их приватно — кандидат на `textops.py` при следующем заходе.
- reply.py всё ещё ~890 строк при целевых ~200 — пайплайн-механика (guard-цикл, стадии,
  отправка) просится на дальнейшую нарезку.
